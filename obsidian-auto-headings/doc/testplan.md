# obsidian-auto-headings 测试计划（testplan）

> 本文档是这个 Addon 的**测试场景清单与真值表**：把「在什么状态下、做了什么操作、应当得到什么结果」
> 一条条写清楚，**既驱动自动化单元测试（dev_tests），也驱动用户手动实测（user_tests）**。
>
> 它存在的根本原因：**很多 bug 单测一个动作时看不出来，只在"特定操作序列"下才暴露**——尤其是
> 「文件里已经有编号了，用户又去改模板的某个字段，下一次触发才出问题」这类**状态转移**场景。
> 这正是用户实测反复踩到的坑（如「默认 → 把序号样式改成中文『一』→ 再把标题间隔符填『、』→ 得到
> `一、一 标题`」），故本文档把**操作序列**当作一等公民来枚举。

## 0. 怎么用这份文档

### 0.1 三类读者

| 读者 | 看什么 | 怎么用 |
|------|--------|--------|
| **开发期的 Agent** | 「当前状态」列里**非 ✅** 的行 | 标 ❌ 的是**已知 bug**，应在 `tests/dev_tests/` 补一条**会失败**的回归测试、再改代码使其通过；标 🔲 的是**还没写测试**的场景，补测试。 |
| **手动实测的用户** | 「操作步骤」+「预期结果」 | 在真实 Vault 里照步骤点一遍，对照预期；不符就是 bug，记录到 `doc/log.md` 或反馈给 Agent。 |
| **写规格的 Agent** | 整篇 | 改规格（`spec.md`）时回头核对：新规格有没有让某条预期变化、有没有制造新的状态转移坑。 |

### 0.2 与 `tests/` 两个目录的关系

- `tests/dev_tests/*.test.ts` —— **自动化**单元测试（Vitest，`npm test` 跑的就是它）。本文档每条标
  **dev** 的场景，最终都应在这里有一条对应断言；行内的 `→ dev_tests/xxx.test.ts` 标注它该落在哪个文件。
- `tests/user_tests/*.md` —— **可复制粘贴进 Obsidian 实测**的样例文档。本文档每条标 **user** 的场景，
  应在这里有一个（或合并进一个）`.md`，文件内先文字写清「场景 + 预期」、再用代码块给可复制内容。
- 一条场景**可以同时**是 dev + user（引擎逻辑能单测、用户也值得手验）。

> **新增/修改功能时的硬性流程**（见 `doc/log.md` 顶部强制规则与本仓库 `CLAUDE.md` §5）：
> 1. 在本文档相应分类下**加/改场景行**，写明预期与当前状态；
> 2. 给 dev 场景补 `tests/dev_tests/` 断言、给 user 场景补/更 `tests/user_tests/` 样例；
> 3. 全绿后再提交。**修了一个 bug，就把对应行的 ❌ 改成 ✅ 并留下根因备注。**

## 1. 核心理念：状态转移测试（本文档的重点）

写编号引擎时最容易只测「干净输入 → 一次编号」，但真实使用是一条**时间线**：

```
空白/裸标题 ──编号──▶ 已编号A ──改配置──▶ ？──再次触发──▶ 已编号B ──又改配置──▶ ……
```

每个箭头都是一次「触发」，每次触发 = **先剥离旧前缀、再写新前缀**。**绝大多数诡异 bug 都出在
「剥离」这一步**：旧前缀是用**旧配置**写的，而剥离器拿的是**新配置**——若剥离器认不出旧前缀，
旧前缀就残留，新前缀叠在它左边，于是 `一、一 标题`、`第1 1 标题`、`1-1 1.1 子` 这类叠加就出现了。

**因此，凡是"模板里任何一个字段可被用户改动"，都要测它的状态转移**：
对每个字段，构造「用配置 X 编号 → 把该字段改成 Y → 再触发」，断言结果**幂等且只剩一层正确前缀**。

> **幂等性总断言（适用于几乎所有场景）：** 对同一内容**连续触发两次**，第二次结果必须**等于**第一次
> （`renumber(renumber(x)) === renumber(x)`）。本文档大量 ❌ 都违反它。dev_tests 应有一个通用 helper
> 跑「随机/枚举配置 → 编号 → 改一个字段 → 再编号 → 断言无叠加」。

> **「2024 折中」（0.3.18 起的剥离语义，见 E5/E5b；0.6.4 彻底解决）：** 无状态的纯文本引擎**无法区分**「`1 ` 是插件写
> 的前缀」与「`2024 ` 是用户写的正文」——两者形状相同（`数字+空格+文字`）。强行二选一会矛盾：若保留
> `2024 总结` 不剥，则 `1 总结` 也不会被剥而叠成 `1 1 总结`（违背幂等）。折中方案：**剥离只剥最左侧的
> 一个完整前缀单元，其后内容一律当正文保留**。于是 `2024 总结`**首次**仍被剥成 `1 总结`（既定取舍），
> 但用户在序号后补回 `2024` 得到的 `1 2024 总结` 会**稳定保留**。两项工程支撑：① 段间「序号间隔符」的
> 容差类**排除空格**（空格只当「序号→标题」分隔符，故 `1 2024` 不被解析成两段序号）；② **不再循环
> 重剥**（一次只移除一个单元）。**0.6.4 根治**：`buildPrefix` 在每个前缀末尾追加 Word Joiner（U+2060，
> 不可见字符），`stripPrefix` WJ 快速路径精确截断，彻底消除歧义——`1 ⁠2024 总结` 中 WJ 是精确边界，
> 剥除后得 `2024 总结`，再触发仍得 `1 ⁠2024 总结`，完全幂等。旧格式（不含 WJ）仍由正则路径向后兼容。

> 前缀也能剥净，不再叠加。

> **已修（B2/B3/B9，0.3.18，方案 A + 只剥一层）：** `stripPrefix` 剥离前后缀时不再死扣当前模板值，
> 而是用 `affixAlternation` 接受一个**候选集合**——恒含「当前级别值 + **空串**」，并由 main.ts 注入

> 同时把剥离改为**只剥一层**（见 §1「2024 折中」）：`1 2024 标题` 只剥 `1 `、保住用户的 `2024`。

> 插件前缀"的部分。

## 4. 随机序列压测（UVM 风格状态转移框架）★

状态转移 bug 的组合是**爆炸**的（文本 × 模板每个字段 × 操作顺序），手写穷举不现实。故引入借鉴硬件验证
**UVM** 的**约束随机序列**框架（`tests/dev_tests/uvm/`，入口 `tests/dev_tests/random_sequence.test.ts`）：
随机生成「编辑文本 / 改模板 / 触发编号」的长序列，用**记分板**自动判对错，并以**功能覆盖率**确认真撞到了
关心的场景。它随 `npm test` 默认跑 500 条 × 60 步（<2s），重型用 `npm run test:fuzz`。

**0.6.2 升级（本轮）★：** 针对「发现潜在 bug 能力不够强」做了三类增强，并即时用它撞出 §3.2 的 U1/U2/U3：

1. **更真实的「打字」激励**：新增 `editTitleInPlace`（在**已带编号前缀**的标题行里继续改文本、旧前缀仍留行上，
   而非旧的 `retitle` 把整行清空重打）与 `mutatePrefix`（手动删字符 / 去空格 / 改数字破坏前缀区）——覆盖
   用户「在 md 里怎么打字、怎么删」的真实主线，这正是 strip 最易出错处。
2. **放开约束、加大空间**：默认序列步数 40→60；新增 explore 模式（`AAH_FUZZ_MODE=explore`）放开**字母样式 /
   inherit×非空前后缀 / 脏标题（分隔符·数字·字母起头）/ 手动破坏前缀**等全部约束。
3. **新增第二记分板「幂等性」**：`renumber∘renumber === renumber`（见 §1 幂等性总断言），**恒成立、与配置无关**，
   故能容纳放开的约束与脏激励而不因 E5/L1 既定取舍误报。它专逮「**再触发就变样**」的多次施加侵蚀——
   **U1 正是它逮到的**（旧的「裸文档参考模型」是单次施加等价性，结构上看不见多次侵蚀，见 §3.2 注）。

两记分板**互补**：默认模式（参考模型，常绿）守受约束的干净空间、能逮残留/叠加；explore 模式（幂等性）在
放开的脏空间里找新 bug。explore 默认 `it.skip`、不进 CI（它会撞到 U1/U2/U3）；修好后转正即成回归网。

**核心不变量（oracle）：** 维护「裸文档真值 `bare`」与「编辑器文本 `rendered`」两份锁步状态，每次触发后断言

```
join(rendered)  ===  renumberContent(serialize(bare), 当前模板)
└ DUT：对带历史前缀的文本剥离+重编      └ 参考：对裸文本直接编号
```

相等 ⟺ 旧前缀被剥得干干净净。**任何叠加/残留当场被抓**（B1–B5、C3 都逮得到），且参考侧复用可信的
build 路径、不会和 DUT 一起错。失败抛 `SequenceError`（含 **seed + 操作轨迹 + 三方文本**），照种子可
100% 复现（`AAH_FUZZ_SEED=N AAH_FUZZ_RUNS=1 ...`）。

**约束 = 当前 strip 健壮性的精确刻画**（每条**默认模式**约束对应一个已登记的 bug/取舍，故 CI 常绿；
explore 模式则把约束全放开、改用幂等性记分板专门找 bug）：

| 约束 | 对应 | 状态 |
|------|------|------|
| ~~`prefix`/`suffix` 整条序列固定~~ → 现在「空 ↔ 候选」随机切换，并传并集给剥离 | B2/B3 | ✅ 已放开（0.3.18）|
| ~~回避数字/字母起头标题~~ → 现在恒喂全部标题（含 `2024 总结`） | L2 | ✅ 已放开（0.3.18）|
| ~~`inherit` 仅当前前后缀都空时才翻转~~ → 现在可在非空前后缀下翻转 | B8 | ✅ 已放开（0.6.2，参考记分板 20000×80 全绿，B8 确认无 bug）|
| ~~`topLevel` 只减不增~~ → 现在可双向随机，升高后两侧一致 | C3 | ✅ 已放开（0.6.0）|
| 默认模式随机样式只用 arabic/cjk/circled（不混字母/罗马） | L1 / U3 | 仍约束（explore 模式已放开，撞出 U3，幂等记分板）|
| 默认模式不用脏标题 / 不破坏前缀区 | E5 / U1 / U2 | 仍约束（U1/U2 已修；explore 模式仍放开探索幂等性）|

> **关键纪律：放开约束 = 扩大覆盖。** 每修好一个对应 bug，就放开对应约束让框架自动覆盖更大空间；
> 放开后若变红即说明修得不彻底。0.3.18 放开 B2/B3、L2 两条后，15000×80 全绿、覆盖率闭合。

> 框架已实战验证：bring-up 阶段在 1.5 万条随机序列里先后撞出栅栏失衡、L2、L1 三类，逐一定位；
> 0.3.18 又在放开 B2/B3、L2 约束后压 1.5 万条确认无回归；**0.6.2 升级 + explore 模式在 20000×80 里撞出
> U1/U2/U3 三类新问题**（见 §3.2），并确认 B8 无 bug（放开其约束后参考记分板仍全绿）；**0.6.3 修复
> U1/U2、新增罗马数字样式，5000×80 全绿**（含 fuzz 验证循环剥离幂等性与 tolerantInnerSeparator）。详见
> `tests/dev_tests/uvm/README.md`。

## 5. 维护本文档的工作流程

1. **加功能 / 修 bug 前**：先在本文档对应分类**写下新场景行**（操作 + 预期 + 初始状态标 🔲/❌）。
2. **写测试**：dev 场景 → `tests/dev_tests/`；user 场景 → `tests/user_tests/*.md`（先文字说明、再可复制
   代码块）。两目录的样例都应能**追溯回本文档某条 ID**。
3. **改代码使其通过**；`npm test` / `npm run lint` / `npm run format:check` 全绿。
4. **动了编号引擎（`src/numbering.ts`/`parser.ts`）时，额外 `npm run test:fuzz` 压一遍**（§4 UVM 框架）；
   修好某个已登记 bug（B2/B3/C3/L1/L2…）后，**放开 `tests/dev_tests/uvm/framework.ts` 里对应的约束**，
   让随机序列自动覆盖新空间——放开后仍绿才算修彻底。
5. **回填状态**：把该行的 🔲/❌ 改成 ✅，补根因/备注；若是 bug 修复，同时更新 §3 汇总表。
6. **按 `CLAUDE.md` §4 维护记忆**：`doc/log.md` 追加交接块、`doc/status.jsonl` 插一行 + 更新首行；
   `spec.md` 若涉及规格变化同步更新；bump 版本号（含 manifest/package/versions/release）。

> **写场景的要诀：** 永远多想一步"**之前是什么状态**"。同样一个"把序号样式改成中文"，在**空文件**上
> 做没问题，在**已编号文件**上做就可能炸——后者才是用户的真实路径。**默认给每条配置变更场景都配一条
> "先用旧配置编号、再改、再触发"的状态转移用例。**
