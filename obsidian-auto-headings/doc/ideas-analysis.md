# obsidian-auto-headings Ideas & Feedback 整理

> 收集于 2026-07-02，汇总用户需求、bug 发现与设计优化建议。
> 分为「M7 冲刺」与「M8 功能迭代」两个阶段执行。

## 发现的问题（优先修复）

### P0 标题文本改动后，反向链接未自动更新

**现象**：
- 文件 A 标题从 `## 细目甲` 改为 `## 细目甲改名字测试`（只改正文，不改编号）
- 文件 B 有链接 `[[A#1.1.1 ⁠细目甲]]`
- 改后链接仍然指向旧标题，未自动更新

**影响**：这是 0.7.6 backlink 同步的**遗漏场景**

**优先级**：⚠️ **M7 冲刺必补**（不该等 M8）— 既然 0.7.6 已做编号同步，就该连标题文本改动一起处理

**设计建议**：
- 捕获「编号前后的标题文本变化」（不仅仅是编号前缀）
- 对比「旧标题 → 新标题」，若不同则生成更新清单
- 可用**简化的更新**（不需要审阅模式，直接改）因为：
  - 源于正文的标题改动，意图明确
  - backlink 引用的就是这个标题，改标题 = 改引用目标

---

### P1 移动版：模板更新后，部分打开文件未同步编号

**现象**：
- 修改模板中数字样式（例如改为中文）
- 某个已打开的文件的编号更新了
- 另一个文件的编号没更新（不确定是否也打开）

**可能原因**：
- 触发检测只对「当前活跃文件」生效？
- 移动版性能限制，没有全局重排？

**优先级**：🟡 **M7 调查** — 可能是移动版 bug，需要复现与诊断

**建议**：
1. 在 iOS 上复现场景
2. 检查 `onEditorChange` 触发机制是否只覆盖当前文件
3. 如非移动版专属，可能是「文件编辑缓冲」导致的

### P2 子树白名单后续标题的计数器行为不一致

**现象**：
```
## 二、 方法
### 2.1 数据来源
### 2.2 实验设置
#### 2.2.1 硬件
#### 2.2.2 软件
## 三、 结论
### 3.1 
#### 3.1.1 
##### 3.1.1.1 
###### a 
# 附录              ← 白名单（子树）
## 图标              ← 子树下，不编号
## 表格              ← 子树下，不编号
# 参考文献           ← 新 H1，计数器重置？
## 四、 我的参考文献  ← 编号延续前面的「三」？还是独立重新开始？
```

**设计问题**：当用「子树」白名单豁免整块时，这个块**是否应该充当计数器重置边界**？

**当前行为**：子树不占计数器槽位，但也不重置 — 其后的标题继续沿用前面的计数

**用户疑问**：这和「H1 作重置边界」的逻辑不一致。应该：
- **方案 A（保留现状）**：白名单块不重置；视为「从计数器中暂时抽离，但不打断流」
  - 优点：逻辑简单，白名单 = 「这块标题不编号」
  - 缺点：用户如果在「附录」这样的大块结构放白名单，后续重新编号会奇怪

- **方案 B（新增开关）**：模板级选项「子树白名单是否重置计数器」
  - 优点：灵活，满足两类用户
  - 缺点：增加配置复杂度

**优先级**：🟡 **M7 设计决策** — 需要在 M7 确定语义，免得上架后用户吐槽

**建议**：
1. 调查「子树 + 后续编号」的真实用例
2. 按绝大多数书写习惯（你说的）做 default
3. 如确需灵活性，M8 再加开关

## Ideas 优先级与执行计划

### ✅ M7 冲刺（2–3 周内完成，1.0 发布前置）

#### 1️⃣ **标题文本 backlink 同步**（新增）

> 原本归为 M8，现因 P0 问题上移。

**描述**：编号或标题正文改动时，自动更新反向链接中的标题部分

**设计**：
```
renumberingProcess 中：
1. 对比「旧标题 → 新标题」
2. 若文本改动，记录映射关系
3. 通过 metadataCache 反查引用方
4. 逐行更新链接
5. 单一事务写回
```

**配置**：沿用 backlink 的现有开关（`enableBacklinkSync`）— 同一个功能，同一个权限

**测试**：
- [ ] 标题正文改、编号不改 → 链接更新
- [ ] 编号改、标题文本不改 → 链接更新
- [ ] 都改 → 链接同时更新
- [ ] 别名场景 `[[#旧|显示文本]]` → 仅改锚点不改别名
- [ ] 跨库链接是否支持（可能暂不支持）

**具体改动**：

##### TAB: 全局设置
- 语言下拉（自动/中文/English）
- 全局自动编号开关 + 命令双向同步
- 防抖延迟滑块 + 重置按钮
- **Backlink 同步开关 + 警告文案**（开关时弹 modal 提醒备份）

##### TAB: 路径模板
- **子 TAB / 折叠**：「路径规则」与「模板定义」（或并排两列）

##### TAB: 关于
- 插件版本
- 一句话描述
- GitHub 链接、Issue 提交链接
- 未来可加「Buy me coffee」链接

#### 3️⃣ **清除全库时禁用全局编号**（M7 交互改进）

**实现**：
```typescript
async clearAllNumbering() {
  const wasEnabled = this.settings.autoNumberingEnabled;
  
  // 禁用全局编号
  this.settings.autoNumberingEnabled = false;
  await this.saveSettings();
  
  // 执行清除...
  // （全库遍历每个 .md 文件，剥离前缀）
  
  // 恢复至操作前状态
  this.settings.autoNumberingEnabled = wasEnabled;
  await this.saveSettings();
  
  // Notice
  new Notice(`已清除全库编号。全局自动编号已${wasEnabled ? '恢复启用' : '保持禁用'}。`);
}
```

**测试**：
- [ ] 全局编号开 → 清除 → 全局编号仍开
- [ ] 全局编号关 → 清除 → 全局编号仍关

#### 4️⃣ **模板预览优化与文本简化**（M7 UX 改进）

**改动点**：

##### 4.1 预览栏只显示一个示例
```
当前：
级别 │ 前缀 │ 序号 │ 预览
─────┼──────┼──────┼──────────────────
H2   │      │ 中文  │ 一、 二、 三、 四、  ← 太多了
H3   │      │ 阿拉伯│ 1.1 1.2 2.1 2.2 2.3 ← 冗余

改为：
H2   │      │ 中文  │ 一、 二、    ← 仅前两个
H3   │      │ 阿拉伯│ 1.1 1.2    ← 仅前两个
```

**优点**：节省面板宽度（特别是移动版），避免预览过长

##### 4.2 表格形式保留，但加横向滑动支持
- 在桌面版：如果表格宽度超出容器，自动显示滚动条
- 在移动版：允许左右滑动看到隐藏的列

##### 4.3 文本说明简化（当前太详细）

**当前范例**（冗长）：
```
「继承前级」可以开或关闭，决定该级标题的序号前缀
是否包含父级序号。开启时，H3 的编号会拼接 H2 的
序号，得到「1.1」的形式；关闭时，H3 仅呈现自己的
序号，如「a)」。大多数情况下建议开启，除非你的
文档结构特殊...
```

**改为**（简洁）：
```
继承前级：拼接父级序号（开）或仅显示本级（关）
```

**其他文本说明的改进原则**：
- 一句话说清楚，细节留给预览
- 用「举例」代替「长段解释」

##### 4.4 默认模板的显示名国际化

**当前**：模板文件 `default.json` 显示为「默认」（中文）

**改动**：
- 文件名保持 `default.json`
- 显示名跟随语言：中文显示「默认」，英文显示「Default」
- **仅对内置默认模板**；用户创建的模板名字不变

**实现**：
```typescript
// SettingsTab.tsx
getDisplayName(template) {
  if (template.name === "默认" || template.name === "Default") {
    return this.settings.language === 'zh' ? "默认" : "Default";
  }
  return template.name;  // 用户模板保持原样
}
```

## 设计决策待审批

### D1: 子树白名单计数器行为

**问题**：当子树白名单豁免整块标题时，其后续标题应该重置计数器吗？

**选项**：
- **A（现状）**：不重置，继续沿用前面的序号 — 逻辑简单，但「参考文献」这类后追结构会编号奇怪

**建议**：
1. 根据你的「绝大多数书写习惯」直觉选 A 或 B
2. M7 定案，M8 实现（如需要）

### D2: 标题文本 backlink 同步模式

**选项**：
- **快路径**（M7）：自动改，无审阅
- **审阅模式**（M8+）：生成清单，用户批准后改

**建议**：M7 先做快路径（风险低，用户意图明确），M8 按需补审阅
