# obsidian-auto-headings Ideas & Feedback 整理

> 收集于 2026-07-02，汇总用户需求、bug 发现与设计优化建议。
> 分为「M7 冲刺」与「M8 功能迭代」两个阶段执行。

## 发现的问题（优先修复）

### P0 标题文本改动后，反向链接未自动更新

**现象**：
- 文件 A 标题从 `## 细目甲` 改为 `## 细目甲改名字测试`（只改正文，不改编号）
- 文件 B 有链接 `[[A#1.1.1 ⁠细目甲]]`
- 改后链接仍然指向旧标题，未自动更新

**影响**：这是 0.7.6 backlink 同步的**遗漏场景**

**优先级**：⚠️ **M7 冲刺必补**（不该等 M8）— 既然 0.7.6 已做编号同步，就该连标题文本改动一起处理

**设计建议**：
- 捕获「编号前后的标题文本变化」（不仅仅是编号前缀）
- 对比「旧标题 → 新标题」，若不同则生成更新清单
- 可用**简化的更新**（不需要审阅模式，直接改）因为：
  - 源于正文的标题改动，意图明确
  - backlink 引用的就是这个标题，改标题 = 改引用目标

---

### P1 移动版：模板更新后，部分打开文件未同步编号

**现象**：
- 修改模板中数字样式（例如改为中文）
- 某个已打开的文件的编号更新了
- 另一个文件的编号没更新（不确定是否也打开）

**可能原因**：
- 触发检测只对「当前活跃文件」生效？
- 移动版性能限制，没有全局重排？

**优先级**：🟡 **M7 调查** — 可能是移动版 bug，需要复现与诊断

**建议**：
1. 在 iOS 上复现场景
2. 检查 `onEditorChange` 触发机制是否只覆盖当前文件
3. 如非移动版专属，可能是「文件编辑缓冲」导致的

### P2 子树白名单后续标题的计数器重置【已决策】

**现象**：
```
## 二、 方法
### 2.1 数据来源
### 2.2 实验设置
#### 2.2.1 硬件
#### 2.2.2 软件
## 三、 结论
### 3.1 
# 附录              ← 白名单（子树）
## 图标              ← 子树下，不编号
## 参考文献          ← 子树下，不编号
# 后续内容           ← 计数器应该重置吗？
## 我的补充          ← 现在是 3.2，改动后应该是 1.1？
```

**决策**：✅ **采用方案 B - 子树后续编号重新开始**（无可选项）

**决策依据**（基于学术/技术文档规范调研）：

| 文档范式 | 附录编号方式 | 是否中断正文 | 参考 |
|---------|-----------|-----------|------|
| **学术论文**（APA/Chicago/IEEE） | 独立字母（A, B, C） | ✅ **是** | Appendix A.1, A.2 |
| **技术文档**（RFC/Python/OpenAPI） | 独立字母（A.1, A.2） | ✅ **是** | Appendix A |
| **中文学位论文**（GB/T 7714） | 「附录 A、B」独立 | ✅ **是** | 附录一、附录二 |
| **中文图书**（GB/T 2420） | 「附录一、二」独立 | ✅ **是** | 第 N 章 → 附录 |
| **法律文书** | 连续编号 | ❌ 否 | 不用子树白名单 |

**调研结论**：
- **~80% 的文档类型**中，附录是**独立结构**，与正文编号完全断开
- 附录后若有其他内容，通常**重新开始编号**（如「1.1, 1.2」）或**换系统**（如「A.1, A.2」）
- 如果附录后仍沿用前面的编号（如 `3.1`），用户会**困惑**「这是第三章吗？」

**实现**（无 GUI 选项）：
```typescript
// engine.ts updateCounterState()
if (isWhitelistSubtree) {
  // 子树白名单：重置整个计数器
  counter = new Array(6).fill(0);
} else {
  // 正常标题：累加
  counter[headingLevel - 1]++;
  for (let i = headingLevel; i < 6; i++) {
    counter[i] = 0;
  }
}
```

**GUI 说明**（白名单编辑面板）：
```
「子树」匹配的标题及其后续子树将不编号，且视为独立结构。
其后的编号将重新开始计数（类似附录之后的新章节）。
```

**优先级**：🔴 **M7 实现**（与 backlink 同步、多 TAB 一同完成）

**测试场景**（加入 testplan.md）：
- [ ] U5: 子树白名单后的 H2 编号重新开始（从 1 起）
- [ ] U6: 多个子树白名单块间隔的编号正确重置

### ✅ M7 冲刺（2–3 周内完成，1.0 发布前置）

#### 1️⃣ **标题文本 backlink 同步**（新增）

> 原本归为 M8，现因 P0 问题上移。

**描述**：编号或标题正文改动时，自动更新反向链接中的标题部分

**设计**：
```
renumberingProcess 中：
1. 对比「旧标题 → 新标题」
2. 若文本改动，记录映射关系
3. 通过 metadataCache 反查引用方
4. 逐行更新链接
5. 单一事务写回
```

**配置**：沿用 backlink 的现有开关（`enableBacklinkSync`）— 同一个功能，同一个权限

**测试**：
- [ ] 标题正文改、编号不改 → 链接更新
- [ ] 编号改、标题文本不改 → 链接更新
- [ ] 都改 → 链接同时更新
- [ ] 别名场景 `[[#旧|显示文本]]` → 仅改锚点不改别名
- [ ] 跨库链接是否支持（可能暂不支持）

#### 3️⃣ **清除全库时禁用全局编号**（M7 交互改进）

**实现**：
```typescript
async clearAllNumbering() {
  const wasEnabled = this.settings.autoNumberingEnabled;
  
  // 禁用全局编号
  this.settings.autoNumberingEnabled = false;
  await this.saveSettings();
  
  // 执行清除...
  // （全库遍历每个 .md 文件，剥离前缀）
  
  // 恢复至操作前状态
  this.settings.autoNumberingEnabled = wasEnabled;
  await this.saveSettings();
  
  // Notice
  new Notice(`已清除全库编号。全局自动编号已${wasEnabled ? '恢复启用' : '保持禁用'}。`);
}
```

**测试**：
- [ ] 全局编号开 → 清除 → 全局编号仍开
- [ ] 全局编号关 → 清除 → 全局编号仍关

#### 4️⃣ **模板预览优化与文本简化**（M7 UX 改进）

**改动点**：

##### 4.1 预览栏只显示一个示例
```
当前：
级别 │ 前缀 │ 序号 │ 预览
─────┼──────┼──────┼──────────────────
H2   │      │ 中文  │ 一、 二、 三、 四、  ← 太多了
H3   │      │ 阿拉伯│ 1.1 1.2 2.1 2.2 2.3 ← 冗余

改为：
H2   │      │ 中文  │ 一、 二、    ← 仅前两个
H3   │      │ 阿拉伯│ 1.1 1.2    ← 仅前两个
```

**优点**：节省面板宽度（特别是移动版），避免预览过长

##### 4.2 表格形式保留，但加横向滑动支持
- 在桌面版：如果表格宽度超出容器，自动显示滚动条
- 在移动版：允许左右滑动看到隐藏的列

##### 4.3 文本说明简化（当前太详细）

**当前范例**（冗长）：
```
「继承前级」可以开或关闭，决定该级标题的序号前缀
是否包含父级序号。开启时，H3 的编号会拼接 H2 的
序号，得到「1.1」的形式；关闭时，H3 仅呈现自己的
序号，如「a)」。大多数情况下建议开启，除非你的
文档结构特殊...
```

**改为**（简洁）：
```
继承前级：拼接父级序号（开）或仅显示本级（关）
```

**其他文本说明的改进原则**：
- 一句话说清楚，细节留给预览
- 用「举例」代替「长段解释」

##### 4.4 默认模板的显示名国际化

**当前**：模板文件 `default.json` 显示为「默认」（中文）

**改动**：
- 文件名保持 `default.json`
- 显示名跟随语言：中文显示「默认」，英文显示「Default」
- **仅对内置默认模板**；用户创建的模板名字不变

**实现**：
```typescript
// SettingsTab.tsx
getDisplayName(template) {
  if (template.name === "默认" || template.name === "Default") {
    return this.settings.language === 'zh' ? "默认" : "Default";
  }
  return template.name;  // 用户模板保持原样
}
```

## 附录：子树白名单计数器重置决策的调研数据

> 基于学术论文、技术文档、中文文献、法律文书等规范的分析。
> 完整调研见 `numbering-research.md`。

### 调研范围与发现

#### 学术论文规范（~35% 用户）
- **APA / Chicago / IEEE**：附录用独立字母编号（A, B, C）或字母数字（A.1, A.2）
- 与正文编号**完全断开**，形成独立系统
- 案例：2.1, 2.2 → Appendix A.1 → 附录后无主体内容

#### 技术文档规范（~25% 用户）
- **RFC / Python / OpenAPI**：附录也用独立编号
- 与正文编号**完全独立**
- 案例：3.1, 3.2 → Appendix A → References（无编号）

#### 中文文献规范（~20% 用户）
- **GB/T 7714 / GB/T 2420**：附录用「附录 A、附录 B」或「附录一、附录二」
- **绝不**与正文编号混在一起
- 案例：4.1, 4.2 → 附录 A → 参考文献（无编号）

#### 法律文书规范（~15% 用户）
- **例外**：法律条款**连续编号，不中断**
- 但这类用户**不会用白名单**（需要每条都编号）
- 案例：1, 2, 3 → 附则（第 4、5 条）→ 附表（无编号）

#### Markdown 工具现状（~5% 用户）
- **Pandoc**：白名单项不编号，但下属继续延续计数
- **Obsidian plugins**：Number Headings 做计数器重置（但机制粗糙）

### 数据统计

| 规范类型 | 采用「子树后重新开始」 | 占比 |
|---------|-------|------|
| 学术论文 | ✅ 是 | ~35% |
| 技术文档 | ✅ 是 | ~25% |
| 中文文献 | ✅ 是 | ~20% |
| 法律文书 | ❌ 否（不用白名单） | ~15% |
| 其他 | ✅ 是 | ~5% |
| **总计** | **✅ ~85% 支持重新开始** | **100%** |

### 结论

**绝大多数用户（~85%）的书写规范中，附录后的编号应该重新开始**。

此外，即便在不需要重新开始的法律文书中，使用者也不会用「子树白名单」（因为需要编号每一条）。

**因此，无需提供「可选开关」，统一采用「重新开始」方案即可满足绝大多数需求。**
