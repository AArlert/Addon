# obsidian-auto-headings

> 一款 Obsidian 插件，自动为 Markdown 文件中的标题编号，支持完全自定义的多模板与按路径配置——让文件始终整洁一致，无需手动维护。首版仅提供中文界面。

## 1. 背景

Markdown 文档——尤其是长篇技术笔记、会议记录或学术写作——受益于有编号的标题。然而手动维护编号极易出错：在中间插入一节就需要重新给后续所有内容编号。

`obsidian-auto-headings` 通过挂接 Obsidian 的编辑器事件，在用户编辑后经可配置的延迟自动检测标题变化，并将编号前缀直接写入文件。用户可精确定义每一级标题的显示格式（中文数字、法律编号、字母列表等），这些规则可按文件夹乃至单个文件分别配置。

## 2. 需求分析

### 2.1 核心需求

| 编号 | 需求描述 |
|------|----------|
| CR-01 | 在可配置的防抖延迟（默认 300 ms）后检测标题变化，并将编号前缀写入当前活跃的 `.md` 文件 |
| CR-02 | 编号直接写入文件（而非仅注入渲染层） |
| CR-03 | 文件中**第一个 H1** 始终豁免编号，视为文档标题 |
| CR-04 | 后续出现的 H1（不规范但须容错）须降级处理，且分两种模式：**实时编辑**时仅将该行 `#` 改写为 `##`（不触动子树）；**格式化命令**触发时则**级联降级**——`#`→`##` 且其下属标题层级整体下移一级，直至下一个原始 H1 或文件末尾 |
| CR-05 | 修剪空格后与白名单条目**完全匹配**的标题不添加编号前缀，且**不占用计数器槽位**（计数器不累加、不跳号） |
| CR-06 | 白名单为**模板级**配置，随模板生效（路径专属模板覆盖全局默认模板）；所有配置**仅在插件设置 GUI 中管理**，不写入 frontmatter |
| CR-07 | 每个标题级别的显示格式由**模板**定义，模板完全可由用户自定义 |
| CR-08 | 可定义多个具名模板；路径匹配规则将每个文件夹（或文件）映射到指定模板 |
| CR-09 | 插件内部使用纯阿拉伯数字计数器追踪各标题级别；显示格式（序号样式与间隔符）仅在写入文件时才应用 |
| CR-10 | 每次触发时对整个文件重新编号（而非仅修改编辑点附近的区域），确保全局一致性 |
| CR-11 | 提供**面板全局开关**与对应的 Obsidian **全局命令**，可整体启用/禁用自动编号 |
| CR-12 | 支持通过 frontmatter 键 `obsidian-auto-headings`（值仅 `ON` / `OFF`）对单个文件局部控制；非法值忽略 |
| CR-13 | 首版界面仅提供**中文** |

### 2.2 非目标

- 仅注入渲染层（CSS 计数器）——编号必须在原始 Markdown 中可见
- 对文件中第一个 H1 编号
- 生成目录（独立关注点）
- 在 frontmatter 中存储任何**配置**（模板、白名单、路径规则刻意排除，以保持笔记整洁）；frontmatter 仅用作单文件开关
- 多文件批量重新编号（后续版本）
- 多语言界面（首版仅中文，后续版本再考虑）
- Git 感知最小化差异（后续版本）

### 2.3 边界情况与设计决策

| 场景 | 决策 |
|------|------|
| 文件中无 H2 及以上标题 | 插件不写入任何内容，文件保持不变 |
| 白名单标题有子标题 | 该标题跳过前缀，且**不占用计数器槽位**（计数器不累加）；子标题在其父级未变的前提下正常编号 |
| 用户手动编辑了编号前缀 | 下一个防抖周期会将其覆盖为正确值；对前缀的手动编辑是预期行为 |
| 存在多个 H1（实时编辑） | 第一个 H1：不编号、不降级。后续每个 H1：仅将该行 `#` 改写为 `##`，**不**触动其子树 |
| 存在多个 H1（格式化命令） | 第一个 H1：不编号、不降级。后续每个 H1：`#` 改写为 `##`，其下属标题依次下移一级，直至下一个原始 H1 或文件末尾 |
| 标题层级跳跃（如 H2 → H4） | 插件对实际存在的标题编号；缺失的中间层级不实例化计数器 |
| 白名单条目含首尾空格 | 双侧 `trim()` 后比较 |
| 当前文件匹配多条路径规则 | 最具体的规则（文件路径 > 文件夹路径 > 全局默认）优先；并列时列表中靠后的规则胜出 |
| 无任何路径规则匹配当前文件 | 使用全局默认模板（`default.json`，面板显示为"默认"） |
| frontmatter 中 `obsidian-auto-headings` 值非 `ON`/`OFF` | 视为非法，忽略该键，按全局开关行为处理 |
| 代码块内的 `#` 行 | 不视为标题，跳过处理 |

## 3. 功能规格

### 3.1 全局开关与命令

- **面板全局开关**：插件设置面板顶部提供一个"启用自动编号"开关，控制整个插件是否对任何文件生效。状态持久化于 `data.json`。
- **全局命令**：注册 Obsidian 命令"切换自动编号（全局）"，可在命令面板或绑定快捷键调用；它与面板开关**双向同步**。
- **立即重新编号命令**：见 [3.9](#39-触发与防抖)，绕过防抖立即对当前文件执行一次完整格式化（采用级联 H1 降级，见 [3.4](#34-h1-处理规则)）。

生效判定优先级（自上而下，任一为"关"即不处理）：

```
面板全局开关 = 开
        ↓
当前文件 frontmatter obsidian-auto-headings ≠ OFF
        ↓
存在匹配的模板（始终至少命中全局默认模板）
        ↓
执行编号
```

### 3.2 单文件开关（frontmatter）

插件读取——且**仅读取**——一个 frontmatter 键，用于局部控制单个文件：

```yaml
---
obsidian-auto-headings: ON
---
```

- 合法值仅 `ON` 与 `OFF`（建议读取时大小写不敏感）。
- `OFF`：即便面板全局开关为"开"，该文件也不被处理。
- `ON` 或该键缺省：跟随面板全局开关。
- **非法值**（非 `ON`/`OFF`）：忽略该键，按键缺省处理（即跟随全局开关），不报错、不改写。
- 这是插件对 frontmatter 的**唯一**交互；插件绝不向 frontmatter 写入任何配置（模板、白名单、路径规则）。

### 3.3 写入文件模型

插件使用 Obsidian 的 `Editor` API 读取并改写标题行。每次触发周期执行以下步骤：

1. 逐行解析完整文件，识别标题行（以一个或多个 `#` 加空格开头且不在代码块内的行）。
2. 对每行标题的文本剥离已有的自动编号前缀。
3. 执行 H1 降级处理（见 [3.4](#34-h1-处理规则)）。
4. 基于当前模板和计数器状态为每个标题重新计算正确的前缀。
5. 以**单一事务**将所有修改写回编辑器（避免产生过多撤销记录）。

> **为何整文件重写？** 局部重写（仅修改编辑点附近）需要追踪编辑点以下哪些标题受到影响、计数器是否需要重置。整文件重写逻辑更简单、结果更正确，对于任何现实长度的笔记来说性能均可接受。

### 3.4 H1 处理规则

第一个 H1 始终被视为"文档标题"：保留 `#` 不变，不编号、不计数。对于**后续出现的错位 H1**，处理方式取决于触发场景：

| 场景 | 行为 | 理由 |
|------|------|------|
| **实时编辑**（防抖自动编号，用户正在书写） | 仅将错位 H1 行的 `#` 改写为 `##`，**不触动其子树** | 用户可能正打到一半，激进的级联改写会打断输入、产生意外位移 |
| **格式化命令**（"立即重新编号"，对已成型文档整理） | **级联降级**：`#`→`##`，且其下属标题层级整体下移一级，直至下一个原始 H1 或文件末尾 | 文档已写好，目标是产出严格、自洽的层级结构 |

**级联降级（仅格式化命令）执行逻辑：**

```
从上至下扫描文件：

1. 遇到的第一个 `# ` 标题 → 标记为"文档标题"。
   操作：保留 `#` 不变；标记为豁免（不编号，不计数）。

2. 此后每遇到一个 `# ` 标题 → 视为"错位 H1"。
   操作：将该行的 `#` 改写为 `##`。
   该标题的下属范围内（直至下一个原始 `# ` 或文件末尾）
   所有标题层级下移一级：
     ##  → ###
     ### → ####
     依此类推。

示例（原始 → 降级后）：

  # 我的文档          →  # 我的文档          （文档标题，不变）
  ## 第一章            →  ## 第一章
  ### 细节              →  ### 细节
  ### 细节              →  ### 细节
  # 附录               →  ## 附录             （降级）
  ## 小节              →  ### 小节            （下移）
```

> **实时模式对比**：同样的输入在实时编辑下，仅 `# 附录` → `## 附录`，而 `## 小节` 保持 `## 小节` 不变。下次执行格式化命令时才会被级联整理为 `### 小节`。

**编号遍：**
结构改写完成后重新解析文件，视其为不含第二个 H1 的标准文件进行编号。后续所有逻辑均基于改写后的标题层级运行。

### 3.5 内部计数模型

插件维护一个计数器数组 `[c2, c3, c4, c5, c6]`，分别对应 H2–H6（H1 为文档标题，不参与计数）。

规则：
- 遇到第 *n* 级标题时，`c[n]` 加一，所有 *> n* 级的计数器归零。
- 计数器数组与显示格式（序号样式、间隔符）**相互独立**；格式转换仅在写入时进行。
- 白名单中的标题**不占用计数器槽位**——计数器既不累加也不归零，因此**不产生跳号**，且不向文件写入前缀。
- 跨级编号（如 `1.1.1`）由各父级计数器值用模板的**序号间隔符**拼接而成；父级一律以纯阿拉伯数字参与拼接（见 [3.6](#36-模板系统)）。
- 内部计算全程使用纯阿拉伯整数，与模板指定的序号样式无关。

**示例——内部计数器状态 vs. 写入文件内容**（默认模板：序号样式=阿拉伯，序号间隔符=`.`，标题间隔符=空格）：

```
标题序列              计数器状态            写入文件（模板应用后）
──────────────       ──────────────       ────────────────────────
H2                →  c2=1            →    1 标题
H3                →  c3=1            →    1.1 标题
H3                →  c3=2            →    1.2 标题
H2                →  c2=2，c3..=0    →    2 标题
H3（白名单）       →  c3 不变（=0）   →    （不写前缀，不占槽位）
H3                →  c3=1            →    2.1 标题   ← 未跳号
```

> **跨级拼接**：H3 的编号 `2.1` 由当前 `c2`（=2）与 `c3`（=1）以序号间隔符 `.` 拼接而来。即使 H2 的序号样式是中文（如 `二`），父级在 H3 的拼接中仍以阿拉伯数字呈现，从而 H2 显示 `一`、`二` 的同时，H3 仍输出 `1.1`、`2.1`。

### 3.6 模板系统

**模板**是一个具名配置对象，为 H2–H6 各级定义编号的显示格式与该模板的白名单。模板的创建、编辑、删除**完全在插件设置 GUI 中进行**，不涉及任何 YAML 或 frontmatter。

每一级的格式由四个**结构化字段**组成（对应 GUI 中的列）：

| 字段 | 含义 | 示例 |
|------|------|------|
| 前缀（`prefix`） | 序号前的自定义文本，可为空 | `第`、`（`、空 |
| 序号样式（`numeral`） | 本级计数器的呈现形式，下拉枚举 | `arabic`(1)、`cjk`(一)、`circled`(①)、`lower-alpha`(a)、`upper-alpha`(A) |
| 序号间隔符（`numberSeparator`） | 拼接各级父子序号的符号，可完全自定义 | `.`（得 `1.1`）、`-`（得 `1-1`） |
| 标题间隔符（`titleSeparator`） | 完整序号与标题文本之间的文本，由用户自行输入（含可能的尾随 `.` 或空格） | `、`、` `（空格）、`. `（点+空格） |

**这些字段如何组合**——某级标题最终写入的前缀为：

```
前缀 + 父级序号（以序号间隔符拼接）+ 本级序号 + 标题间隔符
```

通过组合这四个字段即可覆盖常见排版需求，例如：

```
前缀  序号样式  序号间隔符  标题间隔符   →   写入效果
────  ───────  ─────────  ──────────       ──────────
（空）  cjk      （不适用）   、            →   一、标题
（空）  cjk      （不适用）   （空格）       →   一 标题
（空）  cjk      （不适用）   . （点+空格）  →   一. 标题
（空）  arabic   .           （空格）       →   1.1 标题
（空）  arabic   .           . （点+空格）  →   1.1. 标题
```

> 注：序号间隔符仅在存在父级拼接时显现（H2 无父级，故 `一、` 不含间隔符；H3 起才出现 `1.1`、`1-1` 中的间隔符）。

**模板文件存储位置：**

每个模板保存为插件文件夹下 `templates/` 子目录中的独立 `.json` 文件。**首次启用插件时**，插件自动创建该目录并写入内置默认模板 `default.json`（在面板中显示为"默认"）。

```
.obsidian/plugins/obsidian-auto-headings/
├── main.js
├── manifest.json
├── data.json               ← 插件设置（全局开关、路径规则、防抖延迟等），不含模板内容
└── templates/              ← 首次启用插件时自动生成
    ├── default.json        ← 内置默认模板（始终存在，显示为"默认"）
    ├── 学术风格.json
    └── 技术文档.json
```

文件名由模板名称经过路径安全化处理后生成（非法字符替换为 `-`，空格保留或替换为 `_`，保证跨平台兼容）。`data.json` 中的路径规则通过模板名称（即文件名去掉 `.json` 后缀后还原）引用对应模板。


**模板文件格式（`.json`）：**

```jsonc
{
  "name": "学术风格",
  "levels": {
    "h2": { "prefix": "", "numeral": "cjk",    "numberSeparator": ".", "titleSeparator": "、" },
    "h3": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " "  },
    "h4": { "prefix": "", "numeral": "lower-alpha", "numberSeparator": ".", "titleSeparator": ") " },
    "h5": { "prefix": "（", "numeral": "lower-alpha", "numberSeparator": ".", "titleSeparator": "） " },
    "h6": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": ". " }
  },
  "whitelist": ["目录", "附录", "参考文献"]
}
```

字段说明：

- `name`：模板显示名（`default.json` 固定显示为"默认"）。
- `levels.h2`…`levels.h6`：每级四个结构化字段——`prefix`、`numeral`、`numberSeparator`、`titleSeparator`（含义见上表）。
- `numeral` 合法枚举：`arabic`、`cjk`、`circled`、`lower-alpha`、`upper-alpha`（首版支持的序号样式，后续可扩展如罗马数字）。
- `whitelist`：本模板生效时使用的白名单条目数组（见 [3.7](#37-白名单系统)）。

**默认模板（`default.json`）的内容**——纯阿拉伯多级点分（即 `1` / `1.1` / `1.1.1` …）：

```jsonc
{
  "name": "默认",
  "levels": {
    "h2": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " " },
    "h3": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " " },
    "h4": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " " },
    "h5": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " " },
    "h6": { "prefix": "", "numeral": "arabic", "numberSeparator": ".", "titleSeparator": " " }
  },
  "whitelist": ["目录", "附录", "参考文献"]
}
```

选用结构化 JSON（而非自由格式字符串）的理由：与 GUI 列一一对应、所见即所得、无歧义解析；TypeScript 原生支持、无需额外依赖、Obsidian 的 `app.vault` 文件 API 可直接读写。

**序号样式枚举：**

| `numeral` 值 | 呈现 | 序列 |
|------|------|------|
| `arabic` | 阿拉伯数字 | 1, 2, 3, … |
| `cjk` | 中文数字 | 一, 二, 三, … |
| `circled` | 带圈数字 | ①, ②, ③, … |
| `lower-alpha` | 小写字母 | a, b, … z, aa, ab, … |
| `upper-alpha` | 大写字母 | A, B, … Z, AA, … |

> 跨级拼接时，**父级一律以阿拉伯数字呈现**，仅本级套用 `numeral` 样式。例如 H2 用 `cjk`、H3 用 `arabic`，则 H2 显示 `一`、`二`，H3 显示 `1.1`、`2.1`。

**默认模板**（`default.json`）始终存在且不可删除，但可编辑。其他具名模板可在设置 GUI 中创建与删除——模板名称会出现在所有需要引用模板的下拉框中。重命名模板时，插件自动将旧文件删除并以新名称写入，同时更新 `data.json` 中所有引用该模板名的路径规则。

**设置 GUI 模板区域布局：**

模板区顶部提供 **[+ 新增模板]** 与 **[删除]** 按钮。所有模板（含"默认"）均可在 GUI 编辑：每个模板各占一行，右侧有一个 **[编辑 ▸]** 按钮，点击后该行**向下展开**行内编辑面板（再次点击折叠）；展开的面板以 H2–H6 五行、每行四列（前缀 / 序号 / 序号间隔符 / 标题间隔符）的形式编辑，并附实时预览。

```
┌─ 模板 ──────────────────────────────────────────────────────────────────┐
│  [+ 新增模板]                                                            │
│                                                                          │
│  默认                                                        [删除]ⁿ⁄ₐ [编辑 ▸] │
│  学术风格                                                    [删除]  [编辑 ▾] │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │ 级别 │ 前缀 │ 序号       │ 序号间隔符 │ 标题间隔符 │ 预览           │ │
│  │ ─────┼──────┼────────────┼───────────┼───────────┼─────────────    │ │
│  │ H2   │      │ [中文   ▼] │  .         │  、        │ 一、 二、 三、 │ │
│  │ H3   │      │ [阿拉伯 ▼] │  .         │ （空格）   │ 1.1  1.2  2.1 │ │
│  │ H4   │      │ [字母a  ▼] │  .         │  )         │ a)  b)  c)     │ │
│  │ H5   │ （   │ [字母a  ▼] │  .         │  ）        │ (a) (b)        │ │
│  │ H6   │      │ [阿拉伯 ▼] │  .         │  .         │ 1.  2.         │ │
│  │                                                   白名单：[编辑…]   │ │
│  └────────────────────────────────────────────────────────────────────┘ │
│  技术文档                                                    [删除]  [编辑 ▸] │
└──────────────────────────────────────────────────────────────────────────┘
  （"默认"模板的删除按钮不可用ⁿ⁄ₐ；其余模板可删除）
```

> 展开的编辑面板内也包含本模板的**白名单编辑器**（见 [3.7](#37-白名单系统)），因为白名单是模板级配置。


### 3.7 白名单系统

白名单是**模板级**配置：每个模板自带一份白名单，随该模板对文件生效（即由 [3.8](#38-按路径配置) 的路径解析决定当前文件使用哪个模板，就用哪个模板的白名单）。白名单条目为**完全匹配**字符串，与剥离现有前缀并修剪空格后的标题文本进行比较。

匹配到白名单的标题：**不写入编号前缀**，且**不占用计数器槽位**（计数器既不累加也不归零，因此不产生跳号）。

**默认模板预填充条目**（用户可在该模板的编辑面板中增删）：
```
目录
附录
参考文献
```

**编辑方式：** 在模板的展开编辑面板内，提供一个白名单输入框——键入词语后按 Enter 即添加为一枚条目（chip），显示在右侧；每枚条目旁有 `x`，点击即删除。

```
白名单：[ 输入后按 Enter…            ]   目录 ✕   附录 ✕   参考文献 ✕
```

> **关于覆盖：** 白名单不再有"全局/文件夹/文件"三层作用域与并集开关。覆盖通过模板覆盖实现——路径专属模板会整体取代全局默认模板，连同其白名单一并替换。

> **配置不入 frontmatter。** 模板、白名单、路径规则均专属于插件设置 GUI 管理。插件对 frontmatter 的唯一交互是读取单文件开关键 `obsidian-auto-headings`（见 [3.2](#32-单文件开关frontmatter)）。

### 3.8 按路径配置

路径规则在插件**设置 GUI** 中管理——一个**全局默认模板**选择器，加上一张可视化表格，每行将一个路径模式映射到所选模板（通过下拉框选择）。某路径命中专属模板时，该模板**整体覆盖**全局默认模板（包含其各级格式与白名单）。

**GUI 布局：**

```
┌─ 路径规则 ──────────────────────────────────────────────────────────┐
│  全局默认模板：[ 默认        ▼ ]                                    │
│                                                                     │
│  [+ 添加规则]                                                       │
│                                                                     │
│  #  │ 路径模式                   │ 模板                            │
│  ───┼────────────────────────────┼──────────────────────────────── │
│  1  │ Projects/                  │ [技术文档   ▼]                  │
│  2  │ 读书笔记/                  │ [学术风格   ▼]                  │
│  3  │ 读书笔记/深度工作.md       │ [简洁风格   ▼]                  │
│                                   ↑ 可拖拽排序                      │
└─────────────────────────────────────────────────────────────────────┘
```

> **路径输入补全：** 路径模式输入框接入 Obsidian 的文件/文件夹建议——键入 `/` 时下拉显示可选的真实路径（与多数 Obsidian 插件的路径选择体验一致），减少手打出错。补全文件夹后，该规则匹配此文件夹及其全部子项。

**解析逻辑：**

1. 收集所有路径模式与当前文件匹配的规则。
2. **最具体的匹配**优先（精确文件路径 > 最长文件夹前缀 > 全局默认）。
3. 并列时，**行号更大**（列表中靠后）的规则胜出，便于通过在末尾添加规则来覆盖已有规则。
4. 若无任何规则匹配，使用**全局默认模板**。


### 3.9 触发与防抖

```
用户在编辑器中输入 / 粘贴 / 删除
        ↓
Editor onChange 事件触发
        ↓
防抖计时器重置（默认 300 ms，可配置范围 50–2000 ms）
        ↓  [计时器到期]
插件检查：全局开关开启？该文件未被 frontmatter 关闭？文件含标题？
        ↓  [是]
完整文件重新编号周期启动（实时模式：H1 仅改本行，不级联）
        ↓
以单一事务更新编辑器内容
```

- 防抖计时器以文件为单位；编辑另一个笔记不会取消当前笔记的待处理更新。
- 若文件在计时器触发前被关闭，待处理的更新将被取消（不向已关闭的文件写入）。
- 命令面板中提供**「立即重新编号」**命令，可绕过防抖立即触发，并采用**格式化模式**（H1 级联降级，见 [3.4](#34-h1-处理规则)）。
- 防抖自动触发采用**实时模式**（H1 仅改本行，不级联），避免打断书写。

## 4. 架构设计

```
┌──────────────────────────────────────────────────────────────────┐
│  main.ts（插件入口）                                             │
│  • 注册编辑器变化监听器                                          │
│  • 管理各文件的防抖计时器                                        │
│  • 全局开关 + 命令；读取 frontmatter 单文件开关                  │
│  • 加载 / 保存设置；首次启用时初始化 templates/default.json      │
└──────┬────────────────────────────┬────────────────────────────┬─┘
       │                            │                            │
┌──────▼──────┐           ┌─────────▼─────────┐      ┌──────────▼──────────┐
│  parser.ts  │           │  numbering.ts      │      │  settings/           │
│  • 从原始   │           │  • 计数器状态机    │      │  SettingsTab.ts      │
│    Markdown │           │  • H1 降级逻辑     │      │  • 全局开关          │
│    解析标题 │           │   （双模式）       │      │  • 模板编辑器        │
│  • 剥离现有 │  ──────►  │  • 应用模板字段    │      │   （行内展开+白名单）│
│    前缀     │           │  • 拼接序号/间隔符 │      │  • 路径规则管理器    │
│  • 识别代码 │           │  • 将前缀写回      │      └──────────────────────┘
│    块边界   │           │    编辑器          │
└─────────────┘           └────────────────────┘

settings/
├── TemplateStore.ts     // 模板文件的增删改查（读写 templates/*.json，目录与默认模板自动创建）
└── PathRuleStore.ts     // 路径 → 模板 的解析（白名单随模板，不单独存储）
```

**存储分层：**

| 存储位置 | 内容 | 读写方式 |
|----------|------|----------|
| `data.json` | 路径规则、白名单、防抖延迟等插件设置 | `plugin.loadData()` / `plugin.saveData()` |
| `templates/*.json` | 各模板定义文件，每个模板一个文件 | `app.vault.adapter` 文件 API（`read` / `write` / `exists` / `mkdir`） |

## 6. 技术栈

| 层次 | 选型 | 理由 |
|------|------|------|
| 语言 | TypeScript | Obsidian 官方推荐；为模板 schema 提供类型安全 |
| 框架 | Obsidian Plugin API | 标准插件接口（`Plugin`、`Editor`、`SettingTab`） |
| 构建 | `esbuild`（通过 Obsidian 示例插件构建脚本） | 快速增量构建；符合社区惯例 |
| 测试 | Vitest | 快速、原生 ESM；无需 Obsidian 运行时即可测试解析器与计数逻辑 |
| 代码检查 | ESLint + `@typescript-eslint` | 捕获常见 TS 错误 |
| 代码格式 | Prettier | 统一代码风格 |
| 国际化 | 首版仅中文，文案内联；预留后续抽取为语言文件的空间 | 降低首版复杂度；多语言留待后续版本 |

## 7. 开发环境搭建

> 前置条件：Node.js ≥ 18，用于测试的 Obsidian Vault

```bash
# 1. 克隆到 Vault 的插件目录
cd /path/to/your/vault/.obsidian/plugins
git clone <repo> obsidian-auto-headings
cd obsidian-auto-headings

# 2. 安装依赖
npm install

# 3. 启动监听构建
npm run dev

# 4. 在 Obsidian 中启用插件
# 设置 → 社区插件 → obsidian-auto-headings → 启用

# 5. 运行单元测试（无需 Obsidian 运行时）
npm test
```

> **热重载**：在开发 Vault 中安装 [Hot Reload](https://github.com/pjeby/hot-reload) 社区插件，每次 `main.js` 重新构建时可自动重载 `obsidian-auto-headings`。
